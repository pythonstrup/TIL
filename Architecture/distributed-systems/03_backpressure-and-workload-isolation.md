# 백프레셔와 워크로드 격리 (3편)

> 1·2편이 **"장애가 나도 결과가 틀리지 않게"(정확성)**였다면, 이번 편은 직교하는 축 —
> **"부하가 쏟아져도 시스템이 안 무너지게"(안정성)**. 멱등성이 완벽해도 백프레셔를 놓치면 시스템은 죽는다.
> 무대: 평소 초당 10건 알림톡 시스템에, 마케팅 이벤트로 5분 만에 100만 건이 큐에 쏟아짐.

## 핵심 결론 (TL;DR)

1. **버퍼는 일시적 폭주(burst)를 흡수할 뿐, 지속적 속도 불균형(rate_in > rate_out)은 못 푼다.** 유한 버퍼는 터지는 시점을 미룰 뿐.
2. 버퍼링은 throughput 문제를 조용히 **latency 문제로 바꿔치기**한다. 늦게 도착한 OTP는 *유효시간이 지나* 못 쓴다 → 1편 "유실"의 소프트한 형태.
3. `rate_out`엔 천장이 있다 — **하류 API rate limit** + **이벤트 루프 용량(2편)**. 천장을 무시하고 밀면 재시도 폭풍으로 *더* 나빠진다.
4. **Redis `maxmemory-policy`는 반드시 `noeviction`** (BullMQ 공식). eviction은 부하 정책이 아니라 **큐를 깨는 데이터 손상 버그**. 부하 떨어내기는 애플리케이션이 *의도적으로* 한다.
5. 백프레셔 대응은 3가지 — **① 인입 제어 / ② 의도적 폐기 / ③ 천장 올리기**. 어느 것도 공짜 아님 → 또 **비용 판단**.
6. 정책을 적용하려면 **먼저 워크로드를 격리(bulkhead)**해야 한다. 안 그러면 가장 막 나가는 트래픽이 가장 중요한 트래픽의 운명을 정한다.

---

## 1. 근본 법칙 — 버퍼는 파도를 받지, 산을 못 깎는다

```
[producer] ──폭증: 초당 수천──▶ [BullMQ 큐 = Redis] ──초당 10──▶ [worker] ──▶ [알림톡 API]
                                       ▲ 여기 쌓인다
```

`rate_in > rate_out`이 **지속**되면 그 사이 큐는 무한히 자란다. 핵심 구분:

| | burst (일시적 폭주) | sustained (지속적 불균형) |
|---|---|---|
| 버퍼로? | ✅ 흡수 가능 (파도) | ❌ 못 막음 (산) — 터질 시점만 미룸 |

**확인 비용 오해 교정:** 백로그가 100만이어도 —
- `waiting`은 Redis **리스트**라 dequeue가 **O(1)**. 백로그 크기와 무관. (단 **priority 큐는 정렬 셋이라 O(log n)** — 대량 삽입 시 느려짐.)
- 주기적 **stalled 체커(2편)**는 `active` job만 훑음(= concurrency에 묶임). 백로그와 무관.

**진짜 터지는 것 둘:**
1. **Redis 메모리** — job마다 데이터+인덱스. 100만 건 = 메모리 압박 → §3의 noeviction 폭발.
2. **지연(latency)** — 새 job의 대기시간 ≈ `백로그 / rate_out`. 100만 / 10 = **~28시간**. OTP가 28시간 뒤 도착 = 가치 0. **버퍼링이 throughput 문제를 latency 문제로 바꿔치기**한 것.

---

## 2. rate_out의 천장 — 무엇이 정하나

`concurrency`를 10 → 1000으로 올려 처리량 100배? **천장이 있다:**

- **하류 API rate limit** (보통 *주된* 병목). 제약 이론: 전체 throughput은 가장 느린 단계가 정함.
- **이벤트 루프 용량 (2편)** — concurrency N = job N개가 **한 이벤트 루프** 위에서 인터리브. 어느 선 넘으면 루프 포화 → **하트비트 굶음 → 가짜 stalled 재발**. concurrency 올렸더니 stalled 터지는 역설.
- DB 커넥션 풀·메모리도 천장.

> ⚠️ 천장을 무시하고 더 세게 밀면 **악화**된다: API 한도 초과 → 429 → 재시도 → 재시도가 부하를 또 늘림 → **재시도 폭풍(retry storm)**. 1편의 재시도가 여기선 *부하 증폭기*.

---

## 3. noeviction — 강제 사항과 그 트레이드오프

> **BullMQ 공식(Going to production):** Redis `maxmemory-policy`는 반드시 **`noeviction`**. Redis가 임의로 키를 evict하면 큐가 정상 동작하지 않는다.

- `noeviction` = 메모리 꽉 차면 키를 지우는 대신 **쓰기 거부**(OOM 에러). 즉 `queue.add()`가 실패.
- 이건 사실 **올바른 백프레셔의 한 형태**(§5 ①) — "꽉 찼으니 못 받아"를 producer에 정직히 알림. **단 producer가 그 에러를 처리해야** 함. 안 하면 1편 "producer 유실"로 전락.

**트레이드오프 — Redis를 캐시로도 쓰면 충돌한다:**

> `maxmemory-policy`는 키별·DB별이 아니라 **인스턴스 전역 설정**이다. (논리 DB를 SELECT로 나눠도 안 갈림.)

| 정책 | 캐시 | 큐 |
|---|---|---|
| `noeviction` (큐 요구) | ❌ 캐시가 안 줄어 메모리 잠식 → OOM으로 둘 다 마비 | ✅ |
| `allkeys-lru` (캐시 요구) | ✅ | ❌ 큐 인덱스 키 evict → **큐 붕괴** |

**→ BullMQ는 캐시와 분리된 *전용 Redis 인스턴스*가 필요.** 기존 캐시 Redis 재활용 불가. 이게 noeviction이 강요하는 인프라 비용.

## 4. delayed job 처리 방식 (공식 확인)

> 문서: delayed job은 즉시 처리되지 않고 **별도 "delayed set"**에 들어가며, 시간이 지나면 일반 job처럼 처리된다. **정확한 시각은 보장 안 됨** — 워커가 얼마나 바쁜지, 같은 시각 예약이 얼마나 많은지에 달림.

- **정렬 셋(sorted set)**, 점수 = 실행돼야 할 타임스탬프 → 항상 정렬.
- 거기서 실행되는 게 아니라, 시간이 되면 **`waiting`으로 승격(promote)** 후 일반 job처럼 처리.
- **고정 주기 폴링이 아님** — 셋의 맨 앞(가장 임박한 시각)만 보고 그때까지 블로킹 대기 → 만기된 것만 승격. 비용은 "지연 총개수"가 아니라 "지금 만기된 개수"에 비례.

**오늘 주제와 직결:**
1. "정확한 시각 보장 안 됨"의 한 원인이 **2편** — 승격 트리거도 이벤트 루프 위 타이머라, **루프 막히면 승격도 늦음**.
2. 승격 후엔 delayed job도 **백로그 줄에 똑같이 섬** → `delay`는 "그 시각 *이후*"를 보장하지 "그 시각에"를 보장 안 함. §1 latency와 한 몸.

## 5. 백프레셔 대응 — 원리적으로 3가지 (eviction은 답이 아님)

> ⚠️ **eviction은 부하 정책이 아니다.** Redis eviction은 "오래된 job"과 "큐 인덱스 키"를 구분 못 해 LRU/random으로 집는 대로 지움 → 큐 붕괴. **load shedding은 Redis가 아니라 애플리케이션이 의도적으로** 해야 한다.

| 선택지 | 무엇 | BullMQ에서 |
|---|---|---|
| **① 인입 제어 (flow control)** | producer를 rate-limit / 큐 깊이 임계 넘으면 enqueue 거부 = 유계 버퍼 | noeviction OOM도 일종의 이것 |
| **② 의도적 폐기 (load shedding)** | 큐 상한 두고 넘치면 정책대로 drop(최신/오래된 것) — **애플리케이션이** | — |
| **③ 천장 올리기 (scale rate_out)** | 워커 수평 확장, **또는 병목 제거 — 공급사 복수화** | — |

- BullMQ `limiter: { max, duration }` = **③의 안전벨트**(가속기 아님). API 한도 *아래로* 스스로를 묶어 429를 피함(한도 닿으면 job을 wait로 되돌림).
- ③ 공급사 복수화의 비용: 복잡도 + **1편 멱등 키 문제** — 공급사가 둘이면 한쪽 키를 다른 쪽이 모름 → A사·B사 중복 발송을 *내가* 추적해 막아야 함.

## 6. 워크로드 격리 (Bulkhead) — 정책보다 먼저 칸을 나눠라

OTP(못 버림·못 늦춤)와 마케팅(버려도 됨)이 **같은 큐**에 섞이면:
- ②"버린다"를 큐 전체에 → OTP도 버려짐.
- ①"인입 막는다"를 큐 전체에 → 마케팅 폭주로 큐가 차서 **OTP enqueue도 거부**.
- 안 버려도 OTP가 마케팅 100만 건 **뒤에 줄 서** 유효시간 초과(§1).

→ **공유 큐에선 부하 정책이 우선순위를 구분 못 한다.** 가장 막 나가는 트래픽이 가장 중요한 트래픽의 운명을 정함.

### 한 줄을 나누는 4가지

| 방법 | 무엇 | 격리 수준 |
|---|---|---|
| **① 우선순위(priority)** | 한 큐 안에서 OTP를 줄 앞으로 | **순서만** — 약함 |
| **② 큐 분리** | OTP/마케팅 큐 따로, 각자 워커·동시성·rate limit·정책 | **부하·용량** — 강함 |
| **③ 워커 풀 분리** | OTP 전용 워커로 **용량 예약** | **처리 용량** |
| **④ Redis 분리** | OTP를 다른 인스턴스로 | **장애 범위** — 최강 |

**①우선순위는 부분적 답 — 함정 셋:**
1. **순서만 바꾸고 용량은 안 줌.** 워커가 전부 마케팅에 묶이면 우선순위 높은 OTP도 슬롯 빌 때까지 대기. 새치기는 줄에서만, 일꾼을 늘리진 않음.
2. **메모리·장애 격리 못 함.** 같은 Redis → 마케팅이 메모리 채우면 noeviction OOM으로 OTP enqueue까지 막힘.
3. **priority는 정렬 셋 → O(log n)** (§1 비용).

→ 우선순위는 "latency 순서"만 풀고 "용량·메모리·장애"는 격리 못 함. SLA가 극단적으로 다르면 **②큐 분리(+③전용 워커, 극단엔 ④Redis 분리)**가 정답. 우선순위는 *비슷한 중요도끼리* 줄 세우는 도구.

### 원리 — 격벽(Bulkhead)

> 성질(중요도·SLA)이 다른 워크로드를 한 자원에 섞으면, 가장 막 나가는 워크로드가 가장 중요한 워크로드의 운명을 정한다.
> 배가 격벽으로 칸을 나눠 한 칸 침수가 전체 침몰로 안 번지게 하듯 — **먼저 칸을 나누고(격리), 칸마다 백프레셔 정책을 따로 건다.**

### 단, 격리는 공짜가 아니다 — 강도 ↔ 비용 다이얼

위 ①→④로 갈수록 격리는 강해지지만 **비용도 같이 커진다.** 특히 ④Redis 분리의 대가:
- **인프라**: 인스턴스 하나 더 = 돈. "안정적으로" 하려면 그 인스턴스도 복제·Sentinel/Cluster로 HA를 붙여야 해 **한 대가 아니라 세트**로 늘어남.
- **운영**: 모니터링·백업·패치·장애 대응 대상이 곱절. 장애 지점도 증가.
- **연결·인지**: 커넥션 풀 분리, 설정 분기, 팀이 담을 토폴로지 증가.

그래서 "나눌수록 좋다"가 아니라 **"어디까지 나눌 값어치가 있나"**다. 비용엔 **두 층**이 있다:

| 층 | 무엇 | 성격 |
|---|---|---|
| **바닥(강제)** | 큐 Redis ≠ 캐시 Redis (§3) | noeviction이 **강요** — 선택 아님, 최소 비용 |
| **추가(선택)** | 핵심 큐 ≠ 일반 큐를 **다른 인스턴스로**(④) | **SLA가 정당화할 때만** 지불 |

대부분은 **②큐 분리(같은 인스턴스, 큐 이름만 나눔)**로 충분하고, ④는 "죽으면 회사가 죽는다" 수준(OTP·결제)에만 얹는다.

> 2편 `lockDuration`(가짜 stalled ↔ 진짜 장애 탐지), 1편 at-least/at-most, 3편 격리 강도 ↔ 인프라 비용 — **전부 한쪽을 얻으면 다른 쪽을 내주는 다이얼**이고, 눈금을 어디 둘지가 엔지니어의 비용 판단이다.

## 최종 — 1·2편과 닫히는 고리

- 1편: **경계마다** 신뢰성이 다시 성립해야 한다 → 3편: 격리도 **워크로드 경계마다** 다시 그어야 한다.
- 1·2편: "유실/중복 중 무엇을 감수할지" → 3편: "어느 트래픽을 버리고 누구를 늦출지" — **둘 다 비용 판단.**
- 공통: 엔지니어링은 *확실성*이나 *무한 용량*을 만드는 게 아니라, **무엇을 비용 들여 지키고 무엇을 감수할지를 경계마다 정하는 일.** 정확성에서든 안정성에서든 같다.
