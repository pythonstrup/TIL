# Props를 굳이 Export하지 않는 이유

- Export 하는 순간 해당 타입을 다른 파일에서 분명히 사용하게 된다.
- `omit`을 통한 필드 생략 등으로 어떻게든 끼워 맞춰서 타입을 사용하기 시작하면 컴포넌트가 변경에 취약해지기 시작한다.
- 어차피 부모 컴포넌트 쪽에서 자식 컴포넌트의 props에 마우스를 올리면 타입이 보인다. 그렇게 확인하는 게 타입스크립트의 철학에 맞지 않을까?
  - 타입스크립트는 자동으로 생성되는 Docs를 제공해준다.

# Props 네이밍에 대한 고찰

- `{컴포넌트명}+'Props'` VS `Props`
- 컴포넌트 안에 선언된 Props는 이미 컴포넌트에 대한 정보를 내포하고 있는 것이기 때문에. 굳이 컴포넌트 명을 붙여야 할까?

# 프론트엔드는 왜 객체지향을 사용하지 않을까?

- 가장 중요한 것은 `비동기 대응`과 `상태 관리`다.
- 프론트엔드에서 말하는 상태는 변하는 모든 것이다.
  - 관리해야 하는 상태가 굉장히 많다. => 객체 지향을 사용할 때, 비동기 대응이 어려울 수 있다.
  - 객체가 상태를 책임지고 관리하는 것이 쉽지 않다. 객체 지향이 추구하는 하나의 역할? 지키기 어렵다.
- 함수형 = 파이프라인을 만들겠다.
